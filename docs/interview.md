# 上传/下载组件面试文档（组件化版）

本文不是“项目汇报稿”，而是“组件能力面试稿”。重点回答：

1. 我封装了什么组件，边界是什么。  
2. 相比传统上传方案，最强优势是什么。  
3. 关键细节如何实现，为什么这样实现。  
4. 能否给出可落地的实现例子。  

## 文档职责

- 负责：组件化能力表达、技术差异点、细节深挖与实现示例。  
- 不负责：完整 API 契约字段表与部署运维手册。  

## 1. 组件化定位（不是项目化叙事）

### 1.1 我封装的核心组件

1. `ZustandFileUpload`：完整上传组件（分片、秒传、重试、自动策略、恢复）。  
2. `SimpleUploadList`：简化上传组件（服务器清单 + 上传弹窗 + 自动刷新）。  
3. `ZustandFileDownload`：下载组件（分片下载、暂停恢复、导出）。  

### 1.2 组件边界（面试必讲）

1. 组件输入：`props`（接口地址、并发参数、回调、UI 模式）。  
2. 组件内部：状态管理（Zustand）+ 持久化（IndexedDB/localforage）+ Worker 调度。  
3. 组件输出：UI 状态、业务回调、可恢复任务行为。  

一句话总结：  
“我把上传/下载从页面逻辑抽离成可配置、可回调、可恢复的组件能力，页面只做接线，不做底层传输细节。”

## 2. 最强优势：与传统上传的差异

### 2.1 差异对比（可直接背）

| 维度 | 传统上传（常见） | 本组件方案 |
| --- | --- | --- |
| 上传模型 | 单请求或弱分片 | 分片 + 秒传 + 补传 |
| 网络中断 | 失败后重传整文件 | 仅重传缺失分片 |
| 页面刷新 | 状态丢失 | IndexedDB 恢复任务继续 |
| 组件复用 | 业务页面内联逻辑 | 标准化组件 + 回调接入 |
| 配置治理 | 分散在页面 | 统一 props，支持默认策略 |
| 集成复杂度 | 每页重复开发 | 简化模式可直接嵌入 |

### 2.2 我们最强的优势（建议主讲）

**最强优势：可恢复的组件内核（Resumable Component Core）**

可直接回答：

“相比传统上传只解决‘传得上去’，我这套组件重点解决‘传输过程可恢复’。  
网络中断、页面刷新、批量上传并发等场景下，任务上下文不丢，恢复时只补缺失分片，不重走整链路。”

## 3. 细节深挖与实现思路

## 3.1 深挖点 A：上传状态机

### 思路

把上传拆成明确状态：`queued -> calculating -> preparing-upload -> uploading -> done/instant/error`。  
状态机比“一个 loading 布尔值”更适合分片、重试、恢复与 UI 展示。

### 价值

1. 回调触发时机清晰（start/progress/complete/error/batch）。  
2. 重试与自动清理不会误伤未完成任务。  
3. 可直接映射进度表格、按钮可用性和异常提示。

## 3.2 深挖点 B：双层并发（文件并发 + 分片并发）

### 思路

1. `fileConcurrency` 控制同时处理几个文件。  
2. `chunkConcurrency` 控制单文件内分片请求量。  
3. 网络波动时动态调参，平衡吞吐与稳定性。

### 价值

避免“并发太高打爆网络”和“并发太低速度差”两种极端。

## 3.3 深挖点 C：刷新后续传

### 思路

1. 上传时持续把文件元信息与分片进度写入 IndexedDB。  
2. 页面初始化时扫描中断任务并恢复为“待上传”。  
3. 自动上传开启时直接继续上传缺失分片。

### 价值

刷新不是灾难点，用户感知是“继续任务”而不是“重来一次”。

## 3.4 深挖点 D：简化模式抗污染

### 思路

新增 `settingsSource`：  
1. `localStorage`：适合完整模式，允许用户保留偏好。  
2. `props`：适合简化模式，强制使用默认自动策略。  

### 价值

彻底解决“在完整模式关闭自动上传后，切到简化模式也被带坏”的问题。

## 3.5 深挖点 E：清单刷新单飞（single-flight）

### 思路

多文件上传成功会连发多次刷新，若并发刷新可能出现乱序覆盖。  
通过“串行队列 + pending 标记”保证同一时刻只有一次刷新执行。

### 价值

最终落盘到 UI 的总是“最后一次最新数据”。

## 4. 举例说明细节实现

## 4.1 示例：简化模式固定自动策略（伪代码）

```tsx
<ZustandFileUpload
  uiMode="simple"
  settingsSource="props"
  autoUpload={true}
  autoCleanup={true}
  cleanupDelay={5}
  chunkSize={1024 * 1024}
  fileConcurrency={2}
  chunkConcurrency={2}
/>
```

说明：  
这里核心不是参数本身，而是“策略来源强制为 props”，从机制上屏蔽历史本地开关污染。

## 4.2 示例：上传成功后清单串行刷新（伪代码）

```ts
if (isRefreshing) {
  queued = true;
  return;
}
isRefreshing = true;
do {
  queued = false;
  await fetchServerList();
} while (queued);
isRefreshing = false;
```

说明：  
这个模式能防止并发回刷乱序，保证列表最终一致。

## 4.3 示例：刷新后恢复中断任务（思路代码）

```ts
const recovery = await recoverInterruptedUploadFiles();
await refreshFiles();
if (autoUpload && !isOffline && recovery.recoveredCount > 0) {
  await uploadAll();
}
```

说明：  
恢复逻辑和自动上传逻辑解耦，便于控制“仅恢复状态”与“恢复并继续传”。

## 5. 面试高频追问（组件视角）

**Q：你封装的是组件还是页面逻辑？**  
**A：** 是组件能力。页面只传 props 和回调，分片、并发、恢复、清理都在组件内部实现。

**Q：你的方案和普通 antd Upload 最大差别？**  
**A：** 重点不在选择文件本身，而在“分片任务可恢复”。断网、刷新、批量并发时仍能保持任务连续性。

**Q：为什么要同时做完整模式和简化模式？**  
**A：** 完整模式用于调参和诊断，简化模式用于业务快速接入；两者复用同一内核，减少维护成本。

## 6. 面试收尾模板（30 秒）

“我做的不是单页上传功能，而是可复用的上传/下载组件能力。  
核心优势是可恢复：分片任务持久化、刷新可续传、失败仅补缺失分片。  
同时我通过 `uiMode + settingsSource` 把复杂能力收敛成简化接入模式，既保留完整能力，也降低业务集成成本。”
